Function:
- `ULyraWeaponStateComponent::ClientConfirmTargetData_Implementation(...)`

Problem explained:
- `HitReplaces` from the server is indexed relative to the full target data list.
- But the client code was comparing against an index that only advanced for `Entry.bShowAsSuccess` hits.

Problematic logic described:
```cpp
int32 HitLocationIndex = 0;
for (const FLyraScreenSpaceHitLocation& Entry : Batch.Markers)
{
    if (!HitReplaces.Contains(HitLocationIndex) && Entry.bShowAsSuccess)
    {
        ...
    }
    if (Entry.bShowAsSuccess)
    {
        ++HitLocationIndex;
    }
}
```

Why this is wrong:
- Marker array includes all hits (geometry + hostile etc.).
- Index only increments for success hits.
- Therefore indices go out of alignment.
- If an invalid geometry hit comes before a valid hostile hit, the hostile hit can be removed incorrectly.

Suggested correct direction:
- Compare `HitReplaces` against the full marker index, not the success-only index.

This matched the user’s bug description that valid hostile hits lose confirmation feedback even though the shot still counts.

## Turn 13
**User:**
`xem cho toi loi nay o dau the first weapon pickup can appear to succeed without becoming the true active or equipped weapon`

**Assistant response:**
Traced pickup -> quickbar -> equip flow and found a likely first-pickup ordering bug.

File identified:
- `Source/LyraGame/Equipment/LyraQuickBarComponent.cpp`

Function identified:
- `ULyraQuickBarComponent::AddItemToSlot(int32 SlotIndex, ULyraInventoryItemInstance* Item)`

Problematic order described:
```cpp
if (Slots[SlotIndex] == nullptr)
{
    if (ActiveSlotIndex == INDEX_NONE)
    {
        SetActiveSlotIndex(SlotIndex);
    }

    Slots[SlotIndex] = Item;
    OnRep_Slots();
}
```

Why this is wrong:
- On first pickup, `ActiveSlotIndex == INDEX_NONE`.
- It calls `SetActiveSlotIndex(SlotIndex)` before the slot actually contains the item.
- `SetActiveSlotIndex(...)` calls `EquipItemInSlot()`.
- `EquipItemInSlot()` reads `Slots[ActiveSlotIndex]`.
- At that moment, slot is still empty.
- Therefore nothing is truly equipped.
- Afterwards the slot gets the item, so it can look like pickup succeeded without actually becoming the true active/equipped weapon.

Suggested intended order:
- Put item into slot first.
- Then set active slot if no active slot existed.

File identified:
- `Source/LyraGame/AbilitySystem/Executions/LyraHealExecution.cpp`

Function:
- `ULyraHealExecution::Execute_Implementation(...)`

Bad line identified:
```cpp
const float HealingDone = FMath::Min(0.0f, BaseHeal);
```

Why this is wrong:
- If `BaseHeal > 0`, this returns `0`.
- If `BaseHeal < 0`, this returns a negative number.
- Then code only applies healing if `HealingDone > 0.0f`.
- Therefore heal never applies properly.

Suggested intended fix:
```cpp
const float HealingDone = FMath::Max(0.0f, BaseHeal);
```

Also compared it with damage execution, which correctly clamps positive damage using `FMath::Max(..., 0.0f)`.

Problematic logic identified:
```cpp
if (bClientShouldActivate && bServerShouldActivate)
{
    ASC->TryActivateAbility(Spec.Handle);
```

Why this is wrong:
- The code computes:
  - `bClientShouldActivate`
  - `bServerShouldActivate`
- But requires both to be true at once.
- In practice, many abilities satisfy one path or the other, not both simultaneously.
- This prevents shared OnSpawn startup from firing correctly.

Supporting evidence:
- `ULyraGameplayAbility_Interact` is an interaction-style ability.
- In `Source/LyraGame/Interaction/Abilities/LyraGameplayAbility_Interact.cpp` its constructor sets:
  - `ActivationPolicy = ELyraAbilityActivationPolicy::OnSpawn;`
  - `NetExecutionPolicy = EGameplayAbilityNetExecutionPolicy::LocalPredicted;`
- Therefore it depends on this shared base startup path.

Also traced where on-spawn activation is triggered:
- `ULyraGameplayAbility::OnGiveAbility(...)` calls `TryActivateAbilityOnSpawn(...)`
- `ULyraAbilitySystemComponent::InitAbilityActorInfo(...)` calls `TryActivateAbilitiesOnSpawn()`
- `ULyraAbilitySystemComponent::TryActivateAbilitiesOnSpawn()` in turn calls `LyraAbilityCDO->TryActivateAbilityOnSpawn(...)`

Likely intended fix suggested:
```cpp
if (bClientShouldActivate || bServerShouldActivate)
{
    ASC->TryActivateAbility(Spec.Handle);
}
```

This was identified as the likely root cause for the regression:
"at least one shared on-spawn ability path no longer starts correctly, so expected interaction-style behavior can fail to come online"
